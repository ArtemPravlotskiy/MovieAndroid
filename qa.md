# Стратегия тестирования проекта MovieAndroid & TMDB-Proxy

Проект представляет собой клиент-серверное приложение, состоящее из Android-клиента (Jetpack Compose, MVVM) и прокси-сервера (`tmdb-proxy`). Ниже представлена стратегия обеспечения качества (QA), чек-листы и актуальный статус автоматизации.

---

## 1. Пирамида тестирования (Реализованная архитектура)

Реализованная структура тестов эффективно разделяет зоны ответственности, изолируя бизнес-логику от UI и сетевых зависимостей.


UI / Instrumented Tests (~15% от общего покрытия кода)
Проверка Compose-экранов, Навигации и Диалогов на эмуляторе.

Unit Tests (~76% строк ViewModel, ~62% строк Model)
Тестирование состояний бизнес-логики и моделей данных в JVM.

* **API Tests (Интеграционные):** Проверка контрактов между `tmdb-proxy` и внешним API (TMDB), а также корректность проброса заголовков и кодов ошибок (401, 429, 500) на Android-клиент.
* **Unit / Архитектурные тесты:** Изолированное тестирование бизнес-логики (`ViewModels`, `Repositories`) на чистой JVM с использованием `MockK` для отсечения Android-зависимостей.
* **UI / Instrumented тесты (Jetpack Compose):** Герметичные тесты критических путей пользователя (экраны поиска, деталей, настроек) на эмуляторе/физическом устройстве с использованием `composeTestRule` и `mockk-android`.

## 2. Чек-листы тестирования & Статус автоматизации

### А. Модуль: Поиск и отображение фильмов (Критический путь)
| ID | Описание тест-кейса | Тип | Статус | Комментарий |
| :--- | :--- | :--- | :--- | :--- |
| А-1 | Успешный поиск фильма по названию. Отображение карточек (название, постер, рейтинг). | Позитивный | **Автоматизирован** | Покрыто в `SearchScreenTest` |
| А-2 | Поиск по пустому запросу или спецсимволам (корректная обработка, отсутствие краша). | Негативный | **Автоматизирован** | Валидация во `ViewModel` |
| А-3 | Фильм не найден (отображение экрана "Ничего не найдено", отсутствие бесконечной загрузки). | Негативный | **Автоматизирован** | Проверка пустого стейта `searchResult` |
| А-4 | Переключение в режим **AI-поиска (векторного)**. Изменение плейсхолдера на «Опишите фильм...» и появление индикатора. | Позитивный | **Автоматизирован** | Интеграционный тест `SearchScreenTest` |
| А-5 | Потеря сети во время ввода (отображение Error State, кнопка "Повторить"). | Edge-case | **Автоматизирован** | Покрыто изолированным `ErrorScreenTest` |

| :--- | :--- | :--- | :--- | :--- |
| Б-1 | Добавление фильма в избранное с экрана деталей (смена иконки, обновление в локальной БД). | Позитивный | **Автоматизирован** | Покрыто тестами `SettingsViewModel` |

### В. Модуль: Настройки приложения (`SettingsScreen`)
| ID | Описание тест-кейса | Тип | Статус | Комментарий |
| :--- | :--- | :--- | :--- | :--- |
| В-1 | Смена темы оформления (Light/Dark). Триггер метода `selectTheme` во ViewModel. | Позитивный | **Автоматизирован** | Покрыто в `SettingsScreenTest` |
| В-2 | Изменение языка приложения или масштаба текста (вызов `activity?.recreate()`). | Позитивный | **Автоматизирован** | Логика отслеживается на уровне `SettingsViewModel` |

### Г. Модуль: API `tmdb-proxy`
- [ ] **Позитивный:** Эндпоинты отдают `200 OK` и валидный JSON, соответствующий контракту Android-приложения.
- [ ] **Негативный:** Невалидный/истекший API-токен (возврат `401 Unauthorized` клиенту вместо падения сервера).
- [ ] **Негативный:** Превышение лимитов (Rate Limiting) или падение TMDB (возврат `500` или `429` с понятной ошибкой).

## 3. Метрики покрытия кода (Code Coverage)

На текущий момент локальные Unit-тесты показывают высокий уровень покрытия критически важной бизнес-логики приложения:

* **`viewModel` пакет:** **76% строк кода** (82% классов) полностью защищены от регрессий.
* **`model` пакет:** **62% строк кода** покрыты тестами.
* **Инструментальные UI-тесты:** Успешно проходят **12 из 12** комплексных тест-кейсов на физических устройствах/эмуляторах.
